# Overview  
Audiophile is a modern B2C e-commerce platform specializing in premium audio devices (headphones, speakers, and earphones). The platform solves the problem of providing a seamless online shopping experience for audio enthusiasts while giving sellers a comprehensive admin panel to manage products, orders, and website content. Built as a full-stack Next.js application with a robust database backend, it supports both authenticated users and guest checkout, ensuring accessibility and convenience for all customers.

The platform is valuable because it combines:
- A beautiful, responsive frontend optimized for all device sizes
- A complete database-driven backend with proper inventory management
- Role-based access control for admin and consumer users
- Real-time stock reservation system to prevent overselling
- Comprehensive order management and tracking capabilities

# Core Features  

## 1. Product Catalog Management
- **What it does**: Displays audio products organized by categories (headphones, speakers, earphones) with detailed product pages including images, descriptions, features, and included items
- **Why it's important**: Core functionality that drives sales and helps customers make informed purchasing decisions
- **How it works**: Products are stored in PostgreSQL database with support for multiple image sizes (mobile, tablet, desktop), related products, and comprehensive metadata. Products can be marked as "new" and include detailed specifications

## 2. Shopping Cart System
- **What it does**: Allows users to add/remove products, edit quantities, and persist cart state across browser sessions
- **Why it's important**: Essential for converting browsing into purchases
- **How it works**: Currently uses localStorage for persistence, migrating to database-backed cart with 30-minute stock reservations to prevent overselling. Supports both authenticated users and guest sessions

## 3. Checkout Process
- **What it does**: Handles order creation with form validation, payment method selection (e-money or cash on delivery), and order confirmation
- **Why it's important**: Final step in the purchase funnel, must be secure and user-friendly
- **How it works**: Validates customer information, calculates totals (subtotal + $50 shipping + 20% VAT on products), creates order records, and reserves stock. Supports both logged-in users and guest checkout

## 4. Admin Panel
- **What it does**: Provides authenticated admin users with tools to manage products, view/update orders, upload media, and configure site settings
- **Why it's important**: Enables business operations and content management without technical knowledge
- **How it works**: Protected routes with role-based access control (Better Auth), CRUD operations for products, order status management, and media library management

## 5. User Authentication & Authorization
- **What it does**: Secure login/registration system with role-based access (admin vs consumer)
- **Why it's important**: Protects admin resources and enables personalized experiences for consumers
- **How it works**: Better Auth integration with PostgreSQL, supporting email/password and OAuth providers. Sessions managed securely with proper expiration

## 6. Inventory Management
- **What it does**: Tracks product stock levels and prevents overselling through reservation system
- **Why it's important**: Critical for maintaining customer trust and preventing order cancellations
- **How it works**: Stock reservations created when items added to cart (30-minute expiry), actual stock deducted on successful checkout. Automatic cleanup of expired reservations

## 7. Order Management
- **What it does**: Tracks orders through lifecycle (pending ‚Üí processing ‚Üí shipped ‚Üí delivered) with order history for users
- **Why it's important**: Enables order fulfillment and provides transparency to customers
- **How it works**: Order records with denormalized product snapshots (for historical accuracy), status tracking, and customer information storage

## 8. Media Management
- **What it does**: Stores and serves product images in multiple sizes and types (product, category, gallery, related)
- **Why it's important**: Optimizes performance and provides appropriate images for different contexts
- **How it works**: Images stored in Vercel Blob or Cloudflare R2, URLs stored in database with metadata (size, type, order)

# Technical Architecture  

## System Components

### Frontend Layer
- **Framework**: Next.js 16 (App Router) with React 19
- **Styling**: Tailwind CSS 4 with custom design system
- **UI Components**: Radix UI primitives + shadcn/ui components
- **State Management**: React hooks + localStorage (migrating to database)
- **Image Optimization**: Next.js Image component with Vercel Blob/R2
- **Animations**: GSAP for page transitions and scroll animations
- **Form Handling**: React Hook Form with Zod validation

### Backend Layer
- **API Framework**: Elysia.js (Bun runtime) with OpenAPI documentation
- **Database**: PostgreSQL 16 (local via Docker, production via Neon)
- **ORM**: Drizzle ORM with TypeScript type safety
- **Authentication**: Better Auth with session management
- **File Storage**: Vercel Blob Storage or Cloudflare R2 for product images

### Infrastructure
- **Runtime**: Bun (development) / Node.js (production)
- **Hosting**: Vercel (frontend + API routes)
- **Database**: Docker Compose (local) / Neon PostgreSQL (production)
- **Storage**: Vercel Blob or Cloudflare R2
- **CI/CD**: Vercel automatic deployments with database migrations

## Data Models

### Authentication Models (Better Auth)
- **users**: User accounts with roles (admin/consumer)
- **sessions**: Active user sessions
- **accounts**: OAuth and email/password accounts
- **verifications**: Email verification and password reset tokens

### E-commerce Models
- **categories**: Product categories (headphones, speakers, earphones)
- **products**: Product catalog with pricing, stock, descriptions
- **product_images**: Multi-size, multi-type image storage
- **product_includes**: Items included in product box
- **related_products**: Product recommendations (self-referencing)
- **orders**: Order records with customer and payment info
- **order_items**: Individual line items in orders (with price snapshots)
- **stock_reservations**: Temporary stock holds for cart items (30-min expiry)

### Key Relationships
- Categories ‚Üí Products (one-to-many)
- Products ‚Üí Images, Includes, Related Products (one-to-many)
- Users ‚Üí Orders (one-to-many, optional for guest checkout)
- Orders ‚Üí Order Items (one-to-many)
- Products ‚Üí Stock Reservations (one-to-many)

# Risks and Mitigations  

## Technical Challenges

### Risk 1: Database Migration Complexity
**Challenge**: Migrating from static JSON to database while maintaining frontend compatibility
**Mitigation**: 
- Maintain API contract compatibility
- Use feature flags to gradually migrate
- Comprehensive testing before deployment
- Rollback plan if issues arise

### Risk 2: Stock Reservation Race Conditions
**Challenge**: Multiple users adding same product to cart simultaneously could cause overselling
**Mitigation**:
- Use database transactions for stock operations
- Implement optimistic locking
- Reserve stock immediately on cart add
- Validate stock availability at checkout time
- Cleanup expired reservations regularly

### Risk 3: Better Auth Integration Complexity
**Challenge**: Integrating Better Auth with existing structure and custom roles
**Mitigation**:
- Follow Better Auth documentation carefully
- Test authentication flows thoroughly
- Implement role checks at middleware level
- Have fallback authentication method if needed

### Risk 4: Image Storage Migration
**Challenge**: Moving from local assets to cloud storage (Vercel Blob/R2)
**Mitigation**:
- Use migration script to upload existing images
- Maintain backward compatibility during transition
- Test image loading performance
- Have backup storage option

### Risk 5: Performance Degradation
**Challenge**: Database queries might be slower than static JSON
**Mitigation**:
- Optimize database queries (avoid N+1 problems)
- Implement appropriate indexes
- Use database connection pooling
- Add caching layer where appropriate
- Monitor query performance

## MVP Scope Definition

### Risk: Scope Creep
**Challenge**: Trying to build too much at once, delaying MVP
**Mitigation**:
- Strictly follow phased roadmap
- Defer non-essential features to post-MVP
- Focus on core user flows first
- Regular scope reviews

### Risk: Over-Engineering
**Challenge**: Building complex solutions for simple problems
**Mitigation**:
- Start with simplest solution that works
- Refactor when needed, not preemptively
- Use proven libraries and patterns
- Code reviews to catch over-engineering

## Resource Constraints

### Risk: Limited Development Time
**Challenge**: Completing all phases within available time
**Mitigation**:
- Prioritize MVP features (Phases 1-3)
- Defer enhancements to post-MVP
- Focus on high-impact features first
- Regular progress tracking

### Risk: Database Costs in Production
**Challenge**: Managed database services can be expensive at scale
**Mitigation**:
- Start with free tier (Neon free tier)
- Monitor usage and optimize queries
- Plan for scaling costs
- Consider self-hosted option if needed

### Risk: Third-Party Service Dependencies
**Challenge**: Vercel, Neon, Better Auth could have outages or changes
**Mitigation**:
- Have backup options identified
- Abstract service integrations
- Monitor service status
- Document migration paths

## Data Migration Risks

### Risk: Data Loss During Migration
**Challenge**: Losing existing product data during migration
**Mitigation**:
- Backup all data before migration
- Test migration on staging environment
- Validate data integrity after migration
- Have rollback procedure ready

### Risk: Incomplete Data Migration
**Challenge**: Some product data not migrating correctly
**Mitigation**:
- Comprehensive seed script testing
- Data validation checks
- Manual verification of migrated data
- Fix script for any missing data

## Security Risks

### Risk: Unauthorized Admin Access
**Challenge**: Admin panel accessible without proper authentication
**Mitigation**:
- Strict role-based access control
- Test all admin routes require authentication
- Regular security audits
- Strong password requirements

### Risk: SQL Injection or Data Breaches
**Challenge**: Vulnerable database queries or exposed sensitive data
**Mitigation**:
- Use ORM (Drizzle) to prevent SQL injection
- Input validation and sanitization
- Environment variable protection
- Regular security updates
- Security best practices in code reviews
